## Context

经过多轮排查，当前 ClaudeTalk 已经能够：
- 成功创建钉钉 AI 卡片
- 成功解包 Claude `stream_event`
- 成功提取用户可见 `text_delta`
- 成功调用 `streamAICard()` 并收到 DingTalk API success

但用户侧仍然报告“卡片空白”或“没有符合预期的流式展示”。这说明问题已经不能只靠 Claude 流式日志判断，更可能与模板字段绑定、更新接口的展示语义，或钉钉客户端的刷新行为有关。为了把 Claude 这层复杂性彻底排除，需要一个完全不依赖 Claude 的固定文本更新实验。

## Goals / Non-Goals

**Goals:**
- 提供一个最小卡片实验入口，创建卡片后按固定时序更新同一张卡
- 验证模板是否显示创建文本
- 验证模板是否响应后续更新文本
- 为“模板字段绑定问题”与“短回复体感不流式问题”提供明确分界

**Non-Goals:**
- 不修正 Claude 流式正文解析逻辑
- 不调整卡片节流器或 token streaming 体验
- 不更改钉钉卡片创建和更新的核心协议，只复用现有实现

## Decisions

### 1. 实验必须完全绕过 Claude

**选择**: 实验入口直接调用：
- `createAICard("第一阶段：创建成功")`
- 延迟后 `streamAICard("第二阶段：更新成功")`

**替代方案**: 继续使用 Claude 生成固定文本

**理由**: 只要 Claude 仍在链路中，就会继续引入 thinking、text_delta、节流时机和 provider 行为等变量，无法把问题收敛到模板显示契约。

### 2. 优先提供最小两阶段实验，而不是完整多阶段动画

**选择**: 首先只验证：
- 创建文本是否可见
- 后续更新是否可见

**替代方案**: 一开始就做多阶段 1→12→123→1234 动画

**理由**: 两阶段实验信息密度最高，足以判断“创建可见/更新不可见/全部可见/全部不可见”四类情况。多阶段实验可以作为第二步，而不是第一步。

### 3. 实验结果必须和日志一一对应

**选择**: 为每个实验阶段记录：
- outTrackId
- 发送文本
- 发送时间
- 成功响应

**替代方案**: 只依赖钉钉客户端肉眼观察

**理由**: 只靠肉眼观察无法区分 UI 刷新延迟与真正的调用缺失。日志和 UI 对照后，才能准确判断模板契约问题。

## Risks / Trade-offs

- **[实验入口被误当成正式功能]** → 临时入口可能污染主产品路径。缓解：明确标注为实验/诊断能力，避免默认暴露给普通消息流。
- **[实验结果受模板本身影响]** → 如果模板设计时就不显示对应字段，实验会直接暴露这个问题。缓解：这正是实验目标，不是缺陷。
- **[两阶段实验仍不足以评估“体感流式”]** → 它只能验证创建与单次更新，不足以代表连续流式体验。缓解：先做两阶段验证，再决定是否需要多阶段实验。

## Migration Plan

1. 更新 OpenSpec 工件，定义固定文本卡片实验能力
2. 增加一个可触发实验的入口
3. 复用现有 `createAICard` / `streamAICard` 发送两阶段固定文本
4. 在真实钉钉客户端观察卡片是否从 A 更新成 B
5. 根据实验结果决定后续优先修模板契约还是继续优化流式体验

## Open Questions

- 实验入口更适合放在 CLI 参数里，还是作为一条特殊的机器人命令触发。
- 是否在两阶段实验后再补一个多阶段实验，用于验证钉钉客户端对连续更新的展示效果。
