## Context

当前 ClaudeTalk 的流式卡片链路已经走通了大部分路径：钉钉卡片可创建、Claude `stream_event` 可解包、最终结果文本正确返回。但仍然无法回答一个关键问题：用户看不到明显的流式过程，到底是因为没有中间 `text_delta`，还是因为中间更新已经发出但太晚太快，或者钉钉更新调用没有成功。根因不是纯实现逻辑，而是当前成功路径缺乏可观测性。

现在已有日志只能看到：
- 顶层卡片创建成功
- `stream_event` 被解包成内层事件
- 最终 `replyText` 正确

但仍看不到：
- 每个 `content_block_delta` 的 `delta.type`
- 是否命中了用户可见 `text_delta`
- `onChunk(false)` 是否真的触发
- `throttledUpdate()` 是否立即/延迟发出了更新
- `streamAICard()` 是否成功返回

所以这次变更不是修业务逻辑，而是补一层临时诊断能力，帮助确认真正的阻塞点位于“解析”“中间调用”还是“体感”。

## Goals / Non-Goals

**Goals:**
- 记录中间 `content_block_delta` 的 `delta.type`、文本长度和时间点
- 记录 `onChunk(false)` 与节流器是否真的发起了中间卡片更新
- 记录 `streamAICard()` 成功返回，使中间更新链路可验证
- 让日志足够区分“没有 `text_delta`”“有 `text_delta` 但没发更新”“已发更新但体感不明显”

**Non-Goals:**
- 不改变当前正文提取规则
- 不改变节流时间窗口或卡片模板
- 不修改钉钉卡片创建与更新协议
- 不在本次变更中做任何体验优化，只增加诊断观测

## Decisions

### 1. 诊断日志必须覆盖完整中间链路，而不是只看输入或输出

**选择**: 诊断日志分布在三层：
- Claude 事件解析层：记录 `delta.type` 与提取结果
- 中间聚合层：记录 `onChunk(false)` 与 `throttledUpdate()` 行为
- 钉钉 API 层：记录 `streamAICard()` 成功/失败

**替代方案**: 只在 Claude 事件解析层打日志

**理由**: 只知道 `text_delta` 出现还不够，必须一路确认它是否真的走到了钉钉更新接口。否则仍然无法判断问题在解析层还是调用层。

### 2. 成功日志采用低成本摘要而非完整文本

**选择**: 诊断日志只记录文本长度、时间戳、请求类型和必要的响应摘要，不输出完整正文内容。

**替代方案**: 把每次更新的完整正文都打印到 stderr

**理由**: 完整正文日志虽然最直接，但会严重污染运行日志，也可能泄露过多用户内容。长度与事件类型已足够支持定位。

### 3. 诊断能力默认临时使用，但以稳定字段组织

**选择**: 即使后续会降级或移除部分日志，本次也使用稳定的前缀和字段名，例如：
- `[claude] [stream] delta.type=...`
- `[card] throttled update len=...`
- `[card] stream update success len=...`

**替代方案**: 随意插入零散调试语句

**理由**: 当前问题已经跨越多轮排查。稳定的日志格式更便于下一轮比对不同运行结果，也更适合在真实聊天中快速筛选。

## Risks / Trade-offs

- **[日志噪音增加]** → 中间流式事件可能非常多。缓解：只记录关键字段，不打印完整正文；必要时只对 `text_delta` 和首次/关键路径打日志。
- **[短回复仍然体感不流式]** → 即使日志证明技术流式成立，短回复场景的体验仍可能不明显。缓解：本次目标是先证明真实链路行为，再决定是否做体验改造。
- **[误把诊断当正式能力]** → 长期保留过多调试日志会降低可维护性。缓解：在提案中明确这是诊断能力，后续可按结果裁剪。

## Migration Plan

1. 更新 OpenSpec 工件，定义中间链路诊断要求
2. 在 `src/index.ts` 的流式解析和节流逻辑中增加摘要日志
3. 在 `src/dingtalk.ts` 的 `streamAICard()` 成功路径增加摘要日志
4. 用真实短回复场景验证日志，判断中间更新是否真实发生
5. 基于日志结果决定后续是修逻辑还是改体验

## Open Questions

- 如果日志证明中间更新链路完全成立，后续是否应另起提案，专门解决“短回复体感不流式”的交互问题。
