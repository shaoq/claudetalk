## Context

`streaming-card-output` 已将钉钉 AI 卡片创建路径接入消息回复流程，但当前卡片创建实现仍沿用早期占位式封装：调用方只传入 `chatId`，`createAICard` 再将其直接写入 `openSpaceId`。这在实际钉钉卡片接口中不成立，因为私聊和群聊场景需要不同的空间标识和投放模型。与此同时，当前代码中的请求类型定义、请求体字段和成功状态判定也存在不一致，导致问题不仅体现在 `param.openSpaceIdInvalid`，还会影响后续调试和稳定性。

这个问题横跨 `src/index.ts`、`src/dingtalk.ts` 和 `src/types.ts` 三个模块，且涉及外部 API 协议适配，因此需要在实施前先固定设计边界。

## Goals / Non-Goals

**Goals:**
- 让私聊和群聊场景都能生成符合钉钉要求的 AI 卡片创建请求
- 明确卡片创建所需的最小上下文，并从消息回调一路传递到钉钉客户端
- 统一请求类型定义、运行时请求体和成功判定逻辑，降低后续维护成本
- 为卡片投放失败保留足够的错误信息，便于定位真实的钉钉接口问题

**Non-Goals:**
- 不改变 `streamAICard` 的流式更新协议
- 不重构 Claude CLI 流式输出逻辑或会话持久化逻辑
- 不在本次变更中扩展新的卡片模板能力或富媒体卡片场景

## Decisions

### 1. 用结构化上下文替代单一 `conversationId`

**选择**: 为卡片创建引入明确的投放上下文，至少包含 `conversationType`、`conversationId`、`senderStaffId`，并由 `index.ts` 在收到钉钉回调时传给 `createAICard`。

**替代方案**: 继续只传 `chatId`，在 `dingtalk.ts` 内部猜测当前场景。

**理由**: 卡片投放所需的信息天然来自入站回调，且私聊场景必须依赖 `senderStaffId`。继续只传单一字符串会让协议适配逻辑隐式化，并重复制造“会话 ID 可复用到所有 API”的误解。

### 2. 按场景统一生成 `openSpaceId` 与投放模型

**选择**: 在 `dingtalk.ts` 中集中封装卡片创建请求构造逻辑：
- 私聊使用 `IM_ROBOT` 场景的 `openSpaceId`
- 群聊使用 `IM_GROUP` 场景的 `openSpaceId`
- 请求体同时携带与场景匹配的 open space model 和 deliver model

**替代方案**: 仅修正 `openSpaceId`，暂不补投放模型字段。

**理由**: 这次报错虽然首先暴露在 `openSpaceId`，但官方示例显示卡片投放不仅依赖空间 ID，还依赖场景化投放模型。只修一个字段会留下下一轮接口校验失败的隐患。

### 3. 让类型定义反映真实接口，而不是历史猜测

**选择**: 更新 `src/types.ts` 中的 AI 卡片创建请求类型，使其字段名和嵌套结构与实际发送请求保持一致，并补充卡片投放上下文类型。

**替代方案**: 只修运行时代码，不修类型定义。

**理由**: 当前 `AICardCreateRequest` 仍声明 `openConversationId`，而运行时代码实际发送 `openSpaceId`。这种失配会误导后续开发和审查，属于应当一起清掉的技术债。

### 4. 成功判定统一使用 HTTP 成功语义

**选择**: 使用 `response.ok` 结合响应体中的业务错误字段来判断卡片创建是否成功，并在失败日志中保留 HTTP 状态和远端 message/code。

**替代方案**: 继续用固定状态码 `299` 做判断。

**理由**: 钉钉接口的成功响应通常是 2xx，硬编码 `299` 没有可维护性，且会在接口真实成功时制造假失败。

## Risks / Trade-offs

- **[私聊缺少 `senderStaffId`]** → 若回调中没有 staffId，则无法构造私聊卡片空间。缓解：在创建前显式校验并返回可诊断错误。
- **[钉钉接口字段仍有环境差异]** → 不同应用配置可能对投放模型字段要求不同。缓解：记录完整错误码与 message，并按官方最小可用模型实现。
- **[调用链改动影响已有代码]** → `createAICard` 入参变化会波及现有调用点。缓解：只收敛到卡片模式调用路径，保持 markdown 路径不变。
