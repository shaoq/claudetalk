## Context

当前 ClaudeTalk 的固定文本卡片实验已经证明三件事：一是消息链路能够进入卡片模式，二是 `createAndDeliver` 与 `PUT /card/instances` 都能返回成功，三是问题在绕过 Claude 后依然存在。因此，当前“首屏卡片空白”不再是流式解析问题，而是钉钉卡片协议与模板实际契约之间仍未对齐。

现有实现已经对齐过一轮官方最小 payload，但仍采用 ClaudeTalk 自己的卡片运行时模型：创建阶段只保留 `outTrackId`、单聊目标依赖 `senderStaffId` 推导 `openSpaceId`、更新阶段只提交 `outTrackId + cardParamMap`。用户补充说明其模板与 `dingtalk-openclaw-connector` 的流式卡片模板一致，且字段 key 已确认是 `content`。这意味着下一步不应继续猜模板字段名，而应把协议整体与参考实现/官方文档重新对齐。

## Goals / Non-Goals

**Goals:**
- 让 ClaudeTalk 的卡片创建协议与参考实现保持同一语义，尤其是单聊目标识别和卡片实例上下文
- 让卡片运行时在创建成功后持有足够的信息支撑首屏展示、后续更新和最终结束
- 让卡片更新请求显式表达“按 key 更新”的语义，而不是依赖当前最小 PUT payload 的隐式行为
- 让固定文本实验能够区分“模板契约错误”与“协议不兼容”

**Non-Goals:**
- 不在本次变更中修改 Claude CLI 流式事件解析
- 不设计新的卡片模板样式或替换用户当前模板
- 不处理群聊增强能力以外的消息体验优化，例如 ACK 生命周期或节流体验

## Decisions

### 1. 以“参考实现行为 + 官方文档”作为协议对齐基线

**选择**: 实现阶段先核对 `dingtalk-openclaw-connector` 的卡片服务做法，再将 ClaudeTalk 的创建/更新协议收敛到与其等价的运行时语义，同时用钉钉官方接口字段校验请求体合法性。

**替代方案**: 继续只根据当前接口 success 响应做局部猜测式调整。

**理由**: 当前最突出的问题是“接口成功但 UI 空白”。这种情况下仅凭 success/失败日志继续微调单个字段，排障效率很低；参考实现已经与同类模板配合工作，足以作为更强的对照基线。

### 2. 卡片实例模型必须从“只记录 outTrackId”升级为“完整运行时上下文”

**选择**: 创建成功后，运行时对象至少要保留 `outTrackId`、`cardInstanceId`（若响应提供）、投放目标标识、以及后续更新/结束所需的上下文字段。

**替代方案**: 继续只保留 `outTrackId`，其余字段按需临时推导。

**理由**: 现有模型只够支持最乐观路径，无法验证参考实现依赖的实例标识，也无法在“成功创建但首屏空白”场景下继续做精确比对。卡片对象如果缺少协议上下文，后续更新语义就只能靠猜。

### 3. 更新请求显式表达按字段更新，而不是依赖隐式全量覆盖

**选择**: `streamAICard` 的请求体补齐钉钉更新接口建议的更新选项，例如按 key 更新卡片数据的显式配置，并保留最终结束所需的控制语义。

**替代方案**: 继续只发送 `outTrackId + cardParamMap`，把是否刷新交给服务端默认行为。

**理由**: 当前实验的首要问题发生在创建阶段，但正式流式路径同样依赖更新语义。既然要做协议对齐，就不应保留这种“请求被接受但含义不明确”的隐式更新方式。

### 4. 用固定文本实验作为协议回归，而不是继续用 Claude 输出观察

**选择**: 实施后优先回归固定文本两阶段实验，以“阶段一可见、阶段二可更新”为最低通过标准；只有这一标准满足后，才重新评估 Claude 流式体验问题。

**替代方案**: 修改协议后直接回到正式 Claude 流式场景观察。

**理由**: 当前用户已经通过实验把问题收敛到协议层。继续把 Claude 引回验证链路，只会重新引入无关变量。

## Risks / Trade-offs

- **[参考实现依赖的私聊目标标识与当前回调字段不完全同名]** → 需要在实现时核对 `senderId`、`senderStaffId` 与钉钉建卡目标字段的关系；缓解：先以参考实现实际使用方式为基线，再做真实私聊验证。
- **[补齐实例标识后仍然首屏空白]** → 说明问题进一步收敛到模板数据域或模板发布版本，而非协议；缓解：把验证结果沉淀到日志中，确保下一轮排障不再回到创建/update 成功判定。
- **[协议对齐会波及现有卡片运行时类型与调用点]** → `src/dingtalk.ts`、`src/types.ts`、`src/index.ts` 都会联动调整；缓解：先稳定私聊最小链路，再扩展到其他场景。
- **[官方文档与参考实现字段存在差异]** → 需要在合法请求体范围内优先对齐已知可工作的行为；缓解：设计上明确以“官方字段合法 + 参考实现等价语义”为目标，而不是逐字复制任一来源。

## Migration Plan

1. 对照参考实现和官方文档，梳理当前创建/更新协议与目标协议的字段差异
2. 调整 AI 卡片运行时类型，补齐实例标识和目标上下文
3. 重构 `createAICard` 与 `streamAICard` 请求体及响应解析
4. 用真实私聊执行固定文本两阶段实验，验证首屏与更新都可见
5. 在固定实验通过后，再回归正式 Claude 流式卡片路径

## Open Questions

- 参考实现在私聊场景最终使用的是 `senderId`、`staffId`，还是两者组合映射；实现前需要从其卡片服务代码中确认。
- 钉钉更新接口在当前模板场景下是否还要求 `isFinalize` 或其他结束语义字段进入请求体，而不仅仅是本地日志概念。
