## Context

ClaudeTalk 当前已经接入钉钉 AI 卡片创建与流式更新能力，但运行时链路仍不稳定。第一类问题出现在 Claude CLI：当前实现只在卡片模式下切换到 `--output-format stream-json`，却没有附加当前 CLI 版本要求的 `--verbose`，导致流式模式一进入就失败。第二类问题出现在钉钉卡片接口：`createAndDeliver` 即使返回 HTTP 200，也可能在响应体内部以投放结果声明失败；当前代码只检查 HTTP 状态和 `errcode`，会把这类失败误当成成功，导致后续使用 `undefined` 的卡片标识继续调用 `card/streaming`。

这次变更涉及 `src/index.ts`、`src/dingtalk.ts` 和 `src/types.ts` 三个模块，并直接依赖外部 Claude CLI 与钉钉 OpenAPI 的真实协议，因此需要先固定运行时边界，再进入实现。

## Goals / Non-Goals

**Goals:**
- 让卡片模式在当前 Claude CLI 版本下能够稳定进入流式输出
- 让 AI 卡片创建只有在真实投放成功且返回可用标识时才被视为成功
- 让卡片主流程只在拿到有效卡片上下文后才开始流式更新
- 为卡片创建失败、流式更新失败和 Claude 流式失败定义明确的收敛路径

**Non-Goals:**
- 不在本次变更中设计新的卡片模板样式或占位态文案体系
- 不实现消息撤回或 Markdown ACK 生命周期管理
- 不扩展新的消息类型或富媒体流式场景

## Decisions

### 1. 将 Claude 流式模式视为显式运行配置，而不是隐式复用 JSON 模式

**选择**: 在 `callClaude` 中，当存在 `onChunk` 回调时，显式追加当前 CLI 所需的流式参数组合，包括 `--output-format stream-json` 和 `--verbose`。

**替代方案**: 继续仅切换 `output-format`，在运行时遇错后回退到非流式模式。

**理由**: 当前 CLI 已经把流式模式的参数约束外显为启动前校验。与其在失败后兜底，不如在命令构造阶段一次满足约束，保证卡片模式和 Markdown 模式的执行分支从一开始就明确分离。

### 2. 卡片创建成功判定采用“传输成功 + 投放成功 + 标识齐全”三段式

**选择**: `createAICard` 只有在以下条件同时满足时才返回成功：
- HTTP 响应成功
- 响应体顶层没有业务失败信号
- `result.deliverResults` 中至少有一个投放成功结果
- 响应体返回后续流式更新所需的卡片标识

**替代方案**: 沿用“HTTP 200 即成功”或仅检查 `errcode`

**理由**: `createAndDeliver` 的真实失败经常出现在响应体内部的投放结果里，而不是 HTTP 状态码。只有把“投放成功”和“返回可用标识”都纳入判定，才能杜绝 `Created card: undefined` 这种假成功路径。

### 3. 卡片主流程改为 fail-closed

**选择**: `index.ts` 的卡片模式主流程必须在 `createAICard` 成功后才进入 Claude 流式更新；一旦卡片创建失败，停止后续 `streamAICard` 调用，并按策略降级到 Markdown 回复或错误提示。

**替代方案**: 卡片创建失败后继续调用 Claude，待最终结果出来后再决定是否发 Markdown

**理由**: 当前问题的核心不是 Claude 本身，而是卡片生命周期不受控。先跑 Claude 再补救会让错误路径更复杂，也会让用户侧看到重复消息或不一致状态。

### 4. 流式更新依赖的响应结构需要显式建模

**选择**: 为 `createAndDeliver` 和 `card/streaming` 的响应增加明确的类型定义，并在运行时代码中从结构化字段提取卡片标识、投放结果和错误信息。

**替代方案**: 继续用 `any` 或猜测字段名

**理由**: 当前运行时问题已经说明，类型与真实协议失配会直接制造误判。把关键响应结构建模出来，能同时提高成功判定、错误日志和后续维护质量。

## Risks / Trade-offs

- **[Claude CLI 版本继续变化]** → 未来 CLI 可能再次调整流式参数约束。缓解：将流式参数集中在 `callClaude` 一处构造，并在错误日志中保留完整 stderr。
- **[钉钉响应结构在不同租户下略有差异]** → 可能存在顶层 `success` 成功但部分投放失败的情况。缓解：以 `deliverResults` 的逐项结果作为最终判定依据，并保留原始错误消息。
- **[卡片失败后降级带来双消息体验]** → 卡片创建失败时再发 Markdown，用户可能先看到 ACK 再看到错误/结果。缓解：本次只把卡片路径稳定下来，后续再单独优化占位卡片体验。
- **[流式更新标识仍未完全确认]** → 如果钉钉接口实际返回的标识字段与现有假设不同，仍可能阻塞 `streamAICard`。缓解：先在设计中要求“拿不到标识不得进入流式更新”，实现时依据真实响应补齐解析。

## Migration Plan

1. 更新 OpenSpec 工件，明确运行时成功判定与降级要求
2. 修改 `callClaude` 流式参数构造，确保当前 CLI 版本可启动
3. 修改 `createAICard` 的响应解析与成功判定，阻断假成功路径
4. 调整卡片模式主流程，只有在拿到有效卡片实例后才进入 `streamAICard`
5. 用真实日志场景回归验证：卡片创建失败、Claude 流式失败、正常流式成功三条路径

## Open Questions

- `createAndDeliver` 在当前钉钉环境下返回的“后续流式更新标识”是否固定为 `cardInstanceId`，还是嵌套在 `result` 内的其他字段中，需要在实现时用真实响应进一步确认。
- 卡片创建失败时，是否应该立即降级为 Markdown 正式回复，还是先只发送错误提示，由后续占位卡片变更统一体验策略。
