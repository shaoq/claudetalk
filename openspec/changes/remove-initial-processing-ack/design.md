## Context

当前消息处理链路分成两层确认：

- `src/dingtalk.ts` 在收到钉钉 Stream 回调后，会立即回协议级 ACK，避免阻塞 WebSocket 帧循环。
- `src/index.ts` 在真正处理业务消息前，又会通过 `sessionWebhook` 额外发送一条“👍 收到，正在处理...” markdown 消息。

第二层确认不是钉钉协议要求，而是业务层的用户可见提示。现状会让 markdown 模式多出一条无信息增量的消息，也会让 card 模式出现“先 markdown 确认，再出现 AI 卡片”的重复反馈。

## Goals / Non-Goals

**Goals:**
- 删除用户可见的首条“收到，正在处理...”确认消息。
- 保持钉钉 Stream 协议 ACK 与异步处理机制不变。
- 让 markdown 模式直接输出最终回复，让 card 模式直接进入卡片创建与流式更新。
- 清理现有 OpenSpec 文档中把该确认消息当成设计组成部分的描述。

**Non-Goals:**
- 不调整 Claude CLI 调用方式、会话管理或卡片流式更新节流逻辑。
- 不引入新的中间态提示文案。
- 不改变错误处理和最终回复内容格式。

## Decisions

### 1. 仅移除业务层确认消息，保留协议层 ACK

**选择**: 删除 `src/index.ts` 中两处通过 `callback.sessionWebhook` 发送“确认”markdown 的逻辑，保留 `src/dingtalk.ts` 中对 Stream 回调的即时 ACK。

**替代方案**: 连同底层 ACK 一起移除。

**理由**: 协议层 ACK 用于保障回调处理链路稳定，业务层确认消息只是用户可见提示，两者职责不同。当前要去掉的是冗余用户提示，不是网络协议确认。

### 2. card 模式的首个可见反馈改为卡片本身

**选择**: card 模式收到消息后，直接调用 `createAICard("思考中...")` 作为首个可见反馈，不再先发 markdown 确认。

**替代方案**: 保留确认消息，仅在 markdown 模式删除。

**理由**: card 模式已有卡片初始化内容作为处理中状态，占位信息已经足够，再保留 markdown 确认只会制造两套 UI 并存。

### 3. markdown 模式改为单消息回复

**选择**: markdown 模式等待 Claude 生成完成后直接发送最终回复。

**替代方案**: 将“收到，正在处理...”替换为其他更短的提示。

**理由**: 用户诉求是移除这条首个应答，且当前没有证据表明必须保留某种中间消息。最小变更就是直接删除，而不是替换文案。

### 4. 同步修正文档，避免后续实现回滚该行为

**选择**: 更新 `streaming-card-output` 相关 proposal/design/tasks 中关于“先发收到确认”的表述。

**替代方案**: 只改代码，不改 OpenSpec 工件。

**理由**: 现有文档仍把这条确认消息描述成设计步骤，如果不修正，后续按文档继续实现时很容易再次引入该行为。

## Risks / Trade-offs

- [慢响应时缺少即时可见反馈] → markdown 模式下用户会直接等待最终回复；缓解方式是保留底层 ACK，并在 card 模式继续使用“思考中...”卡片作为首个可见状态。
- [历史设计文档与现状不一致] → 如果只改代码不改文档，后续维护会混淆；缓解方式是同步更新相关 OpenSpec 工件。
- [card 创建失败时无预先提示] → card 创建失败回退 markdown 前，用户不会先收到确认消息；缓解方式是保持回退逻辑不变，确保最终仍能返回 markdown 结果或错误信息。
